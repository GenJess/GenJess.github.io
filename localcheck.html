Problem

Pickup basketball is broken.

Not the game — the logistics. Every day, players drive to empty courts, spam group chats, or just skip going because they don't know if anyone will show.

😤
Empty courts
No way to know if anyone's there before you make the drive. Google Maps shows where courts are — not what's happening.
📱
Group chat chaos
50-message threads just to coordinate a run. "Anyone there?" repeated daily with no real answer.
🏆
No competitive layer
Results disappear after games end. No rankings, no history, no accountability. The court has no memory.
📅
Zero planning tools
Nothing exists to say "I'll be at Zilker at 6pm Saturday." Players show up and hope.
Features

Everything the court was missing.

Each feature serves a single interaction loop: Check in → See who's there → Play → Log results → Climb the ranks → Come back tomorrow.

🟢
Real-Time Check-ins
One tap to announce you're at the court. Everyone sees it instantly via Postgres LISTEN/NOTIFY. Auto-expires after 2 hours — no ghost players. Optional status notes ("Running 5s", "Looking for one more").
📅
Schedule — Visual Planner
Full day/week/month calendar where players post when they plan to show up. Interactive time-slot creation, player avatar timeline — reduces the uncertainty that keeps people home.
🏆
ELO Rankings
Every court has its own competitive hierarchy. Log 1v1 results, ELO handles the math. Default 1500. Score tracking, dispute flagging, court-scoped leaderboards. Your ranking follows you.
🏃
Runs — Event Org
Create structured pickup events: format (5v5, 3v3, King of the Court), player cap, RSVP tracking. Auto team generation using ELO-balanced shuffling.
🗺️
Court Discovery
Dark-mode Mapbox map with orange pulsing markers. Add courts that don't exist on Google or Apple Maps. Nominatim geocoding (free, no API key). "Set as My Court" for your home base.
🔔
Real-Time Notifications
Court activity, match confirmations, dispute alerts — all delivered via Postgres change subscriptions. Zero-latency. No polling.
Architecture

The design decisions that matter.

Every choice here was deliberate. Court-scoped data, adapter pattern, real-time via Postgres — not polling.

Why court-scoped everything?
The pivotal call: every piece of data (check-ins, plans, leaderboards, runs) is tied to a court_id. This creates natural micro-communities — your local court's leaderboard is yours. It makes the data model composable and the product feel personal. Changing this later would require rebuilding everything.
Why Postgres LISTEN/NOTIFY instead of polling?
Real-time presence is the core value prop. Polling every 5 seconds would add latency, cost, and complexity. Supabase's real-time subscriptions sit on Postgres change events — instant updates when someone checks in, live leaderboard after match results. The app would feel broken without it.
Why a database adapter pattern (src/lib/db.ts)?
App code never imports Supabase directly — it imports from db.ts. Swapping backends requires changing one file. Business logic stays clean and testable. Type safety enforced at the boundary. This was overkill for a personal project, but it's how production systems should be built.
Why Nominatim for geocoding instead of Google Maps?
Free, no API key, good coverage for court discovery. Courts are sparse enough that premium geocoding accuracy is wasted spend. If coverage gaps emerge in production, swapping to Mapbox Geocoding is a one-line change in the adapter.
Why ELO at default 1500?
Standard chess ELO baseline. Simple to explain to users, well-understood math, graceful convergence. The alternative (TrueSkill, Glicko-2) adds complexity with marginal accuracy gains for a casual pickup context. Good enough is correct here.

Database

18 tables, fully typed.

Every table has RLS policies. Every relationship is enforced at the DB layer. No application-level auth gates that can be bypassed.

profiles

notifications
id · user_id type · payload read · created_at
RLS on every table. Users can only read/write their own records unless explicitly granted access. Leaderboard reads are public within a court. Match writes require both parties to be authenticated. Admin override via service role only.
Lessons

What building this taught me.

🔄
Real-time is a product decision, not a tech one
Adding real-time subs was trivial technically. The hard part was designing the UX so it feels expected, not jarring. When a new check-in appears mid-view, how does it animate in? These are product questions disguised as engineering ones.
🏗️
Schema design is irreversible at scale
The court_id scoping decision touched 12 of 18 tables. Getting it right before writing a single component saved weeks of refactoring. Most PMs never see this — building it taught me why DB design deserves as much time as feature specs.
📱
Web-first, mobile-second is a real tradeoff
Sports apps live on mobile. The web app proved the concept but the adoption ceiling is real. iOS development alongside web doubled the surface area — one backend, two frontends, two different mental models of state management.
🔐
RLS is a product feature, not an afterthought
Designing row-level security forced me to think clearly about data ownership. Who can see what? Who can write what? These are product questions. Building it at the DB layer meant I couldn't be lazy about access control.
